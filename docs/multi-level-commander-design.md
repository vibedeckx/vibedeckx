# 多层级指挥官（Multi-Level Commander）设计

> 状态：**设计探讨**（2026-06-22）。本文记录 Main Chat 从"带工具的聊天框"演进为
> **递归式指挥官体系**的产品思考与架构设计。尚未实现；当前线上只有 workspace 级
> 的 chat orchestrator（见 [`chat-session-orchestrator-state.md`](./chat-session-orchestrator-state.md)
> 与 spec [`superpowers/specs/2026-03-28-terminal-tools-main-chat-design.md`](./superpowers/specs/2026-03-28-terminal-tools-main-chat-design.md)）。

---

## 1. 出发点：两个交互"海拔"

Vibedeckx 的定位是 **"Mission control for your coding agents"**。要兑现这个词，
产品需要两个交互高度：

- **执行层（低海拔）**：用户直接在 agent 窗口里跟某个 coding agent 对话，做具体的事。
  *（已有）*
- **调度层（高海拔）**：用户在 Main Chat 里描述**目标**，由一个"指挥官"去拆解任务、
  调度 agents / executors / terminal / browser 来完成。*（本文设计）*

没有调度层，"mission control" 只是个好看的 dashboard；有了它，Main Chat 才真的是
**控制中枢**。

---

## 2. 核心产品论点：卖的是"对人类注意力的杠杆"

coding agent 时代有个根本变化：**写代码变便宜了，监督变成了瓶颈。** 起一个 agent
几乎零成本，但每个 agent 都要人去：确认它没跑偏、回答提问、批准危险操作、卡住时救它。
**稀缺资源从"代码产能"变成了"人的注意力"。**

整套指挥官体系的价值，一句话：

> **每加一层指挥官，就把更多单位的工作，压缩进同一份人类注意力里。**

这本质上是一套 **selective autonomy（选择性自主）**：给用户委派的杠杆，又不夺走他的
控制权。这是它区别于"放一堆 agent 自己野跑"的根本。

---

## 3. Workspace 指挥官（第一层 / 地基）

这是要先做扎实的一层，上面两层全部复用它的原语。

### 3.1 形态：常驻指挥官（live supervisor）

指挥官不是"派发后退场"的 fire-and-forget，而是**派出后持续监督、汇总、能中途调整**。
用户主要待在 Main Chat 里，指挥官替他盯着下面的 agents/工具。

### 3.2 唤醒模型：事件驱动

常驻 ≠ 让 LLM 一直跑着烧 token。指挥官平时**休眠**，只在"有意义的事发生"时被唤醒、
看一眼、决定动作、再休眠。复用现有事件流（WebSocket + JSON patch、`/api/events` SSE；
agent / executor 运行完都会发事件）。

第一版做两类唤醒，**不做**第三类：

| 唤醒时机 | 说明 | 第一版 |
|---|---|---|
| **① 关键节点** | agent 完成 / 报错 / 卡住 / 发出 `approval_request` 时介入 | ✅ 做 |
| **③ 用户驱动** | 用户在 Main Chat 问进展、要求改方向时，指挥官去查状态、下指令 | ✅ 做 |
| **② 持续语义监控** | 实时盯输出流、发现跑偏就主动打断纠偏 | ⏳ 后续（需一直消费 token 的观察者，性价比待验证） |

> 注：② 的*能力*其实通过 3.6 的"向 session 发消息"原语留好了口子——指挥官被唤醒后
> 可以主动纠偏，只是不做"自动盯梢"那部分。

### 3.3 规划：plan-first（显式计划）

用户给目标 → 指挥官先**拆解成一份可见的子任务清单 / 依赖图** → 用户过目确认 →
然后按图派发、事件驱动地推进、勾掉完成项。

选 plan-first 而非"涌现式"的原因：那份用户确认过的计划，**就是指挥官每次事件唤醒后
重新定位自己的"地图"**。涌现式在多 agent、长时、异步场景下，几乎必然让用户和指挥官
同时失去方向感，且指挥官醒来时重建上下文更难（图是隐式的）。

### 3.4 状态模型：一个 session = 一个 mission

- **`chat_session` = 一次 mission / 指挥官 agent 的持久上下文。**
- 运行一个用户 task = **把该 task 的文本注入到 Main Chat session**。task 只是"喂目标"的
  众多入口之一（手敲是另一个），因此 **Main Chat 不与 `tasks` 表绑定**。
- 一个 session 不会注入多个 task；指挥官自己拆解去做即可。所以**不需要单独的 mission
  实体**——session 本身就是 mission。
- 拆出来的 **subtask 挂在 `session_id` 下**（新存储），构成 session 级的"作战图"。
  指挥官事件唤醒后从这份 subtask 列表重建上下文。

> ⚠️ 区分清楚：现有 `tasks` 表是**用户的任务列表**（用户想做的一个个 to-do），
> **不是**拆解后的执行步骤。subtask 是执行期临时拆出来的，是另一个概念、另一套存储。

### 3.5 执行模型：调度为主、轻活自理（混合）

指挥官**默认是调度者**，但握着工具的轻活自己做：

- **需要一个能自主多步迭代的 coding agent 的子任务** → **派生 agent session**（自己的
  窗口、branch/worktree；干完发完成事件唤醒指挥官）。并行是跨 branch/project 的
  （受"每 branch 最多一个 active session"约束）。
- **executor / terminal / browser 这类指挥官本就握着工具的轻量动作** → 指挥官**自己
  调用工具**直接做，不必专门起一个 agent。

判断标准：**这个子任务需不需要一个能自主多步迭代的 coding agent？需要就派出去，
不需要就自己用工具干。**

### 3.6 指挥官工具箱

复用现有的 `runExecutor` / `getExecutorStatus` / `runInTerminal` / browser，**新增三个
调度原语**：

1. **新建 agent session**——把子目标交给一个全新 agent。
2. **向已有 agent session 发消息**——追加指令、纠偏、串下一步、回答它的提问。
3. **读取某个 agent session 的状态/最新输出**——被唤醒后用它重建"这个 agent 到哪了、
   说了啥"，再决定下一步。（没有它，指挥官只能发不能看。）

> **发消息的现实细节**：目标 agent 可能正忙（每 branch 一个 active session、turn 占用）。
> - 空闲/等输入 → 直接发。
> - 正在跑 turn → 第一版**默认排队**（这轮结束再投喂）；"打断/steering"留给明确的纠偏场景。

### 3.7 信任与批准边界

**层面一（计划确认后，推进子任务要不要逐个问）**：**放养**。计划已过用户的目，
逐步确认会让 B 退化成"换皮的同步问答"。只在 ① 关键节点回报。

**层面二（子 agent 自己冒出 `approval_request` 时谁来批）**：**分级、默认保守**。
- 读类 / 低风险操作 → 指挥官**代批**。
- **破坏性 / 不可逆操作（删除、`rm`、force push、改生产配置等）→ 一律冒泡给用户。**

两个理由：
1. **产品**：用户盯一个指挥官，而不是盯 N 个 agent 窗口里的批准弹窗——这正是常驻
   指挥官比"放一堆 agent 自己跑"更值钱的地方。
2. **安全**：指挥官是个**能派生带权限 agent 的角色**，等于权限放大器。在 `--auth`
   多租户下，无差别代批子 agent 的危险请求是危险的提权面。默认保守、危险操作人来点头。
   （参见 `docs/SECURITY-NOTES.md` 与项目历史中大量 authz 教训。）

## 4. 递归：同一模式套三层

向上扩展时，**这不是三种不同的东西，而是同一套编排模式套了三层**。关键洞察：

> **每一层指挥官的"执行单元"，就是它下一层的指挥官。**

| 层级 | 执行单元 | 主要职责 | 自主程度 |
|---|---|---|---|
| **Workspace** | agent session / 工具 | **执行**：拆解、派 agent、干活 | 高（计划确认后放养） |
| **Project** | 一个个 workspace mission | **协调**：排 `tasks` 优先级、并行调度多 branch、管依赖/合并冲突 | 中（更需人确认计划再 fan-out） |
| **Global** | 一个个 project mission | **分诊 + 路由 + 汇报**：跨项目看哪需要关注、把目标路由给对的 project、出总览 | 低（偏"指方向"，少"亲自做"） |

三层**复用完全相同的三个原语**：建立下级 / 向下级发消息 / 读下级状态。只是"下级"从
agent session，换成 workspace 指挥官、再换成 project 指挥官。事件也一样：
**指令向下派，状态/事件向上冒。**

### 4.1 关键原则：职责"越往上越收敛"

不能三层都做成"自主执行者"，否则 fan-out 失控、成本爆炸、用户彻底失去方向感。
**自主性逐层递减、职责逐层转变：执行 → 协调 → 分诊/路由。** 越往上 blast radius 越大、
目标越模糊，所以越该"帮你看清全局 + 把活派对地方"，而非"自己闷头跨项目执行"。

### 4.2 Project 层的现成抓手

**`tasks` 表（用户任务列表）天然属于 project 层。** 所以 project 指挥官有个特别具体的
职责——**"帮我消化任务 backlog"**：看 task 列表 → 决定先做哪个、能否并行 → 分别在不同
branch 开 workspace mission → 跟踪、汇总、协调合并。这是 workspace 之上最自然、价值最
具体的下一步。

### 4.3 Global 层的定位

价值**类型变了**：不是"执行力"，而是"**态势感知**"。它是那块**总览屏 + 分诊 + 路由**：
哪个项目卡了、哪个 PR 该合了、哪个 backlog 在积压、今天该先看哪儿。它的价值不是替你
跨项目改代码（风险最高、价值最低），而是**让你一个球都不会掉**。这才是 "mission control"
这个词的字面兑现，也是支撑订阅、制造每日打开的东西。

### 4.4 UI 心智：一棵可下钻的树

Global 看到各 project → 进一个 project 看到它的 workspace missions → 进一个 workspace
看到它的 agents。每个节点都有一个"指挥官 chat"，用户 focus 到哪层就跟哪层对话。

## 5. 为什么"递归"本身有额外产品价值

不只是三个独立功能拼在一起，递归结构本身带来：

1. **心智模型只学一次**：每层都是"下个目标 → 看计划 → 确认 → 它执行并升级例外"。
   学会 workspace，project / global 不是新东西。
2. **信任向上复利**：信得过 workspace 指挥官后，把活交给 project 指挥官只是一小步，
   不是重新冒一次险。递归让信任累积，而非每层从零重建。
3. **复刻人本就懂的组织结构**（IC → team lead → manager）。卖给用户的是"一支他来管的
   agent 组织"，而非"一个他来操作的工具"——framing 的溢价完全不同。
4. **优雅升级 = 不在环里也不失控**：问题冒泡到恰当的海拔，琐碎的低层处理掉、重要的
   决策浮到你面前。你得到委派的杠杆，又不丢控制权。

---

## 6. 三盆冷水（价值的前提与风险）

价值真实且大，但**不来自"层数"，来自"每层都真的省了你的注意力"**：

1. **价值 100% 取决于"升级判断"的质量，不是管线。**
   若指挥官动不动升级、什么都 CC 给你，它就是个"凡事抄送你的中层"——**负价值**。
   管线（建 session、发消息、读状态）是体力活；真正的护城河是**指挥官知不知道
   "什么值得打扰你"**。这是产品的生死线，架构图保证不了。

2. **信任是闸门，价值"顺序解锁"——这锁死了 build order。**
   用户不信任下一层之前，上一层对他毫无价值。没法把 global 卖给还不信任 workspace
   的人。所以三层必须**逐层把信任挣到手**，下一层才开始有意义。这是 build order 的
   产品理由——**不是技术难度排序，是信任解锁排序。**

3. **递归只在"问题能在本层解决"时才省注意力。**
   底层问题若需深上下文才能定、却冒泡三层到 global，总览屏也帮不了你——还得一路下钻
   回 workspace 看现场。**漏下去的委派会变成幻觉式的掌控感。** 所以升级判断不光要
   "该不该打扰你"，还要"打扰你时能不能把上下文带够"。

---

## 7. 演进路线图（按"信任解锁"排序，非技术难度）

1. **先把 workspace 指挥官做扎实**（§3）——地基，上面两层全靠它复用。本文 spec 的
   首要落地范围。
2. **再做 project 指挥官**——抓手是 `tasks` backlog + 多 branch 并行，价值最具体。
3. **最后做 global 指挥官**，第一版定位为**"总览 + 路由"而非自主执行**——最贴品牌，
   但应该轻、应该晚。

workspace 的设计必须**为上层递归留好口子**：同一套三原语、事件向上冒、subtask/mission
可选地关联上层 mission（纯做溯源，不产生依赖）。

---

## 8. 待定 / 后续探讨

- **升级判断策略**（冷水一）：指挥官"什么值得打扰用户"的具体规则与可调性——产品生死线。
- **作战图（subtask board）的具体形态**：字段、状态机、依赖表达、UI 呈现。
- **mission 的中断 / 恢复**：长时 mission 跨重启如何持久化与续跑。
- **用户的"目标焦点"问题**：用户更怕"失控"（→ global 态势感知最值钱）还是更烦"琐碎
  监督"（→ workspace/project 注意力压缩最值钱）？这决定 spec 把劲儿往哪使。
- **`--auth` 多租户下的提权面**：指挥官作为权限放大器的边界与代批策略审计。
